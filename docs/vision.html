Product & architecture vision · July 2026

Version control should understand intent, identity, and trust.

A proposal for turning the current Rust prototype into a complete Git replacement: permission-aware at the object layer, friendly to humans and agents, compatible enough to adopt, and opinionated enough to remove Git’s daily tax.

01 · Thesis

Don’t build “Git with ACLs.” Build a trust-aware collaboration system.

Git assumes every clone receives one complete, readable history. Your idea changes that assumption. Once history is actor-specific, the product needs a new storage, sync, merge, release, and audit model—not just new commands.

PRIMITIVE 01

Change

A stable unit of intent owned by a person or agent. It survives edits, automatic snapshots, review, and integration.

PRIMITIVE 02

Snapshot

An immutable content state. Cheap, automatic, encrypted where needed, and never confused with the human story.

PRIMITIVE 03

View

The repository materialized for one identity, capability set, device, and task. There is no universal readable checkout.

PRIMITIVE 04

Line

A protected integration target such as main, release/6.1, or production—not a place where people work directly.

PRIMITIVE 05

Operation

An append-only record of source-control actions. It powers undo, audit, recovery, concurrency, and safe history editing.

PRIMITIVE 06

Release

A signed, immutable projection of a line for a defined audience, with provenance and policy—not merely a tag.

Product position: the system should feel simpler than Git in daily use, while the internals become more rigorous. Engineers think in “my change,” “update it,” “share it,” and “land it.” The engine handles snapshots, ancestry, rebasing, policy, and recovery.
02 · Hard truths

The security promise must be precise.

This product can materially improve coordinated disclosure and secret handling. Overclaiming would make it dangerous.

A public patch is eventually inspectable

If the fixed Linux source is public, attackers can compare the old and new releases even when the private change, message, tests, and review never leak. The system reduces the disclosure window and removes the ready-made roadmap; it cannot make a shipped public fix unknowable.

Revocation cannot erase prior plaintext

If an authorized laptop once decrypted a blob, later revocation cannot prove every copy disappeared. Use short-lived grants, device trust, auditing, watermarking where appropriate, and rotate secrets after access changes.

Metadata leaks

Names, sizes, timing, graph shape, conflict paths, reviewer identity, and “something changed in auth” can all reveal intent. Private payloads need encrypted manifests and padded or deliberately generic public envelopes.

Server-side hiding is insufficient

Access control must gate object transfer and key release. A client must never receive restricted readable bytes and merely hide them in its UI.

03 · User model

One obvious workflow, with complexity pushed into policy.

Keep the jj insight—stable changes plus evolving snapshots—but use vocabulary familiar enough for Git users. “Line” can be shown as “branch” in compatibility mode; the underlying semantics stay clean.

Start a changeIntent and target are explicit
Work normallyAutomatic snapshots protect every state
SharePolicy decides who receives which objects
LandSystem updates ancestry and tests the result
ReleaseSigned projection for a chosen audience

Default CLI

rg change start fix-token-replay --private security
# edit; snapshots happen automatically
rg status
rg share
rg land main
rg release create 6.12.4 --from main --audience public

No commit ceremony

snapshot is an internal durability boundary; change is what humans discuss. “Commit” remains as a compatibility alias that updates the current change and adds a message checkpoint.

That preserves muscle memory without preserving Git’s overloaded object/workflow concept.

04 · Permissioned history

Policy belongs on objects, actions, and derived results.

A safe merge result is not automatically public just because one input was public. Derived-object policy must conservatively include every contributing source until an explicit declassification step approves a public projection.

Identity & capabilities

Users, services, agents, devices, teams, and time-bounded roles receive signed capabilities such as read, materialize, derive, review, integrate, release, administer, and audit.

Policy domains

Reusable domains—public, team/security, env/prod, customer/acme—avoid copying user lists onto every blob. Policy versions are immutable and auditable.

Cryptographic envelopes

Each object is content-addressed over canonical plaintext, encrypted with a random data key, and stored with wrapped keys for authorized policy groups. Manifests and metadata can be separately encrypted.

Policy-safe derivation

Merges, diffs, builds, conflict records, search indexes, caches, and AI context inherit the union of input restrictions unless a trusted declassifier produces a reviewed output.

Selective sync

The sync service performs authorization before object discovery and transfer. Clients receive opaque placeholders or stable redaction envelopes only when policy explicitly permits their existence to be known.

Auditable release

A release is a signed manifest of exactly what an audience may receive. Private provenance can prove the build came from reviewed inputs without publishing those inputs.

Security-fix flow

Private changePatch, exploit test, discussion
Restricted CIEphemeral keys; isolated runners

Environment configuration

Commit the shape, ownership, validation rules, and secret references. For low-sensitivity values, commit encrypted values directly. For high-value production credentials, commit a versioned reference to a secret manager and materialize the value only into an authorized process.

# tracked, reviewable configuration
DATABASE_URL = secret://prod/api/database-url@v7
FEATURE_LIMIT = 250
schema = "url"
owners = ["team/platform"]

Why not a plaintext .env blob?

Repositories replicate and retain history by design. Keys need rotation, expiry, usage policy, and sometimes hardware-backed access. Treating the repo as the control plane is excellent; treating it as the only secret vault is usually not.

The developer experience can still be one command: the workspace broker resolves authorized values without copying them into durable files.

05 · Beyond worktrees

Separate task state from dependency state.

Two agents cannot safely write two different versions to the exact same pathname in one ordinary filesystem namespace. But they can share one repository store, dependency caches, build artifacts, and base tree while receiving isolated writable overlays.

Workspace sessions

A session binds identity + change + view + environment + writable overlay. It has a stable ID, not an exclusive branch checkout. main is a protected line and is never “checked out” or locked by a forgotten folder.

Virtual materialization

Expose a per-session filesystem through overlay mounts, a FUSE/ProjFS-style provider, containers, or tool-aware path virtualization. All sessions share immutable blobs and a read-only base; only changed files consume new space.

Shared warm environment

Package caches, Rust target artifacts, toolchains, indexes, and generated dependencies are mounted from content-keyed caches. A lockfile + compiler + features + target hash selects the cache; it is never blindly shared when writes could race.

Agent boundaries

Each agent receives its own overlay and capability token, even when launched from the same visible project root. A session-aware shell or editor routes each process to the correct view. This prevents cross-agent file clobbering and secret leakage.

The experience: one canonical project folder in the UI, many lightweight sessions underneath. Users see “main,” their changes, and active agents—not scattered clone directories. Under the hood, isolation is real because same-directory concurrent writes without namespaces are not.
06 · Integration

Replace merge-vs-rebase with “land my change.”

Merge and rebase are implementation choices. The user’s intent is usually: preserve my logical change, update it onto the accepted project state, verify it, and integrate it safely.

01 · COMPUTE

Virtual integration

Re-evaluate the change against the current target line without rewriting the user-visible identity. Semantic and file-level merge drivers produce a candidate snapshot.

02 · VERIFY

Policy-aware checks

Run required tests, ownership review, signatures, secret scanning, provenance checks, and restricted CI using only runners authorized for every input.

03 · ADVANCE

Atomic landing

Advance the line only if its expected head still matches. Otherwise recompute. The operation log preserves every prior result; no force-push ritual is required.

Conflicts become objects

A conflict is durable, permissioned, assignable, and resolvable by a person or specialized agent. Unrestricted users see neither private sides nor revealing paths. Resolutions can be recorded as reusable merge rules.

Advanced controls remain

Experts can request topology policies such as linear, merge-preserving, squash projection, or exact parent selection. Teams set the default on the line; individual developers do not debate it on every change.

08 · Architecture

Keep the trusted core small and make policy unavoidable.

The current single-binary JSON store is a good executable specification. The production system should retain understandable objects while replacing filesystem scans, flat JSON, and simulated actors with robust boundaries.

Experience layerCLI, TUI/GUI, IDE protocol, review UI, agent SDK, Git-compatible porcelain
Workflow engineChanges, lines, releases, review, land queue, conflict objects, operation transactions
Policy engineIdentity, capabilities, path/object rules, derivation labels, declassification, audit decisions
Workspace engineFilesystem watcher, overlays, view materialization, environment broker, content-keyed build caches
Graph engineCanonical manifests, DAG queries, diffs, merge drivers, reachability, garbage collection
Crypto & storageHashing, envelope encryption, signatures, key rotation, local object DB, chunking, pack/index format
Sync & hostingAuthorization-before-discovery, resumable transfer, line transactions, policy service, remote audit log

Suggested Rust boundaries

crates/
  rgit-objects      canonical object schemas
  rgit-store        object DB, pack, GC
  rgit-crypto       encryption, signing, key envelopes
  rgit-policy       capabilities and derivation labels
  rgit-graph        ancestry, diff, merge
  rgit-workspace    watcher, overlay, materialization
  rgit-sync         protocol and transactions
  rgit-gitbridge    import/export and remote helper
  rgit-cli          human interface

Non-negotiable invariants

  • Objects are immutable; pointers advance transactionally.
  • Unauthorized clients cannot discover or decrypt restricted objects.
  • Derived data cannot become less restricted implicitly.
  • Every state-changing command is an operation with undo semantics.
  • Releases are reproducible, signed projections—not mutable folders.
  • Format and protocol versions are explicit from day one.
09 · From this repo

Build evidence in layers.

The prototype already demonstrates policy-filtered changes, snapshots, line views, diffs, merges, conflicts, and operations. Keep it as the semantic testbed while each milestone replaces one simulated boundary with a real one.

Harden the local object model

Split the monolith into object/store/graph crates; define canonical serialization and schema migration; add typed markers and release objects; make operation updates transactional; add property and crash-consistency tests.

Prove cryptographic views

Implement real identity, signatures, per-object encryption, key envelopes, metadata redaction, key rotation, and policy-safe derivation. Commission a threat model and external design review before promising secure secret storage.

Make local work irresistible

Add automatic snapshots via a watcher, universal undo, fast status/diff, background maintenance, workspace sessions, shared dependency/build caches, and editor integration. Benchmark on Linux-scale repositories.

Build sync and a self-hosted remote

Design authorization-before-discovery, resumable object transfer, line compare-and-swap, audit replication, partial views, offline capability expiry, backup, and disaster recovery.

Deliver the Git bridge

Losslessly import ordinary Git history, export public/audience projections, add a remote helper, mirror lines to Git hosts, and publish a compatibility matrix. Never export restricted history by default.

Layer on collaboration

Native review, merge queue, CI attestations, agent sessions, ownership, audit UI, release coordination, and enterprise identity. Hosting is where the access-control promise becomes a complete product.

Recommended first vertical slice: two users clone different encrypted views of one hosted repository; a security engineer lands a restricted change into an embargoed line; CI builds it; a signed public release publishes only the approved projection; an auditor reconstructs the full private provenance.
10 · Adoption

Engineers switch for less friction, not a more elegant DAG.

Access-controlled history is the differentiator, but daily delight is the wedge. The product has to be faster, safer, and easier even for teams that never ship an embargoed fix.

Instant safety

Automatic snapshots, obvious undo, no detached HEAD, no lost stash, no destructive rebase fear.

Warm sessions

New agent or task sessions reuse dependencies and builds without sharing writable source state.

One command to land

No recurring merge-versus-rebase decision; policy and automation choose the team’s topology.

Gradual migration

Mirror an existing Git remote, let individuals opt into the client, then adopt native protected views.

Success metrics

  • Time from clone/open to a runnable workspace
  • Status and snapshot latency at monorepo scale
  • Percentage of conflicts resolved before the user sees them
  • Recovery success after mistakes or interrupted operations
  • Restricted-object leakage tests: target must be zero
  • Time to onboard a Git-fluent engineer

Trust strategy

  • Open specification and independently testable format
  • Open-source local client and cryptographic core
  • Threat model, security audits, reproducible releases
  • Export path so customers are never trapped
  • Clear compatibility levels rather than “works with Git”
  • Dogfood on this repository and publish benchmarks
Recommendation

Keep the jj-inspired core. Make the view—not the commit—the fundamental security boundary.

Preserve stable changes, snapshots, lines, conflicts, and the operation log already present here. Add releases, typed markers, real cryptographic policy, session-based workspaces, and a Git bridge in that order. The winning product is not “more powerful Git.” It is a calmer default workflow backed by a permissioned object system Git’s architecture cannot safely emulate.
